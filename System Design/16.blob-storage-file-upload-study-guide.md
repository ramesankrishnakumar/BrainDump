## 1. What is Blob / Object Storage

Object storage holds large binary artifacts — images, videos, documents, backups, ML model weights — behind a flat key like `uploads/user42/avatar.jpg`.

| Layer | What it stores |
|---|---|
| Object store (S3, GCS, Azure Blob) | Raw bytes, keyed by path |
| SQL / metadata DB | Owner, key, timestamps, status, permissions |

The two are always used together: the DB is the source of truth for *what exists*; the object store is the source of truth for *the bytes*.

## 2. The Presigned URL Upload Pattern

### Why not let the client supply the URL?

The naive approach — client uploads anywhere, then POSTs the resulting URL to the server — is untrustworthy. The client could send any URL: someone else's file, a malicious resource, or a path they don't own.

### The correct pattern

```
Client ──(1) request upload──► Server
Client ◄──(2) presigned URL─── Server ──(3) writes pending record──► DB
Client ──(4) PUT file────────────────────────────────────────────► S3 (quarantine)
                                Server ◄──(5) event notification──── S3
                                Server ──(6) update status──────────► DB
```

**Step by step:**

1. Client asks `POST /upload` with file name and content type.
2. Server authenticates, generates a presigned PUT URL with a pre-chosen object key: `uploads/{userId}/{uuid}.jpg`.
3. Server writes to DB: `{ fileKey, bucket: "quarantine", status: "pending" }`.
4. Server returns the presigned URL to the client.
5. Client uploads the file directly to S3 — your server never touches the bytes.
6. S3 fires an event notification (→ Lambda / SQS) when the upload lands.
7. Server confirms the upload and updates `status`.

**What lives in the DB:**

```
fileId     UUID
userId     FK
fileKey    "uploads/user42/avatar.jpg"
bucket     "quarantine" | "prod"
status     "pending" | "clean" | "rejected"
createdAt  timestamp
```

**Key rule: store the object key, never the URL.** Presigned URLs expire — storing them is useless. The key is stable across bucket migrations and region changes.

## 3. Direct Upload vs Proxying Through Your Backend

By default, use direct upload (presigned URL). Proxy only when you need to *see inside* the bytes before they reach storage.

### When to proxy

| Reason | Why it requires proxying |
|---|---|
| Virus / malware scanning | Must read the stream to run AV checks |
| Content validation | Verify it's actually a valid PDF / image, not a renamed binary |
| Transcoding on ingest | Resize images, convert video to multiple formats |
| Encryption with your own keys | Keys S3 doesn't control |
| DLP / compliance | Scan for PII or regulated data before storage |

### When NOT to proxy

- Large media files (video, raw images) — bandwidth cost is brutal
- High-throughput uploads — your servers become the bottleneck
- Anything where S3 Transfer Acceleration or CDN proximity matters

### Middle ground (common production pattern)

Use presigned direct upload for all files, then trigger **async post-upload processing** (Lambda, container worker) for scanning and transformation. You get the benefits of direct upload without skipping validation.

## 4. Enforcing Upload Size Limits

Presigned PUT URLs support a `content-length-range` policy condition baked into the signature:

```python
s3.generate_presigned_url(
    "put_object",
    Params={"Bucket": "quarantine", "Key": file_key},
    ExpiresIn=300,
    Conditions=[["content-length-range", 0, 10_240]],  # 0–10 KB
)
```

If the uploaded object exceeds the limit, **S3 rejects the upload mid-transfer** with a 403. The file never lands.

### The remaining gap

The client still *initiates* a 1 GB TCP transfer before S3 aborts it. You can't prevent the connection from starting. Mitigations:

- Rate-limit presigned URL generation per user
- Short presigned URL expiry (60–300 s)
- WAF rule dropping requests with `Content-Length` over your threshold before they reach S3

## 5. Virus Scanning: The Quarantine Bucket Pattern

S3 cannot scan a file *before* it lands. The standard solution is two buckets and an async scan worker.

### Why two buckets

- Files in quarantine are **never served** to users regardless of scan timing
- If the scan worker crashes mid-scan, the file stays in quarantine, not prod
- IAM policies can block all app/public access to the quarantine bucket entirely

### Flow

```
Client ──► quarantine-bucket
                │
                └── S3 Event Notification
                          │
                          ▼
                       Lambda
                    (AV scan)
                    /         \
               clean         infected
                │                │
                ▼                ▼
        copy → prod-bucket    delete
        delete from quarantine
        DB: status = "clean"  DB: status = "rejected"
                              notify user
```

### Scan options

| Option | Characteristics |
|---|---|
| ClamAV | Open source, run in Lambda or container, free but you manage signature updates |
| AWS GuardDuty Malware Protection | Native S3 integration, managed signatures, priced per GB scanned |
| Third-party SaaS (Cloudmersive, Trend Micro Cloud One) | Better detection rates, pay-per-scan, external dependency |

### The unavoidable window

There is always a gap between upload landing and scan completing. The `status` field bridges this:

- `pending` — uploaded, not yet scanned
- `clean` — scanned and safe, moved to prod
- `rejected` — infected or invalid, deleted

**Never serve a file until `status = "clean"`.**

## 6. Download Flow

```
Client ──(1) GET /files/{id}──► Server
                                Server ──(2) lookup DB──► DB
                                         status ≠ clean → 404
                                         status = clean ↓
                                Server ──(3) generate presigned GET URL
Client ◄──(4) presigned URL──── Server
Client ──(5) fetch file──────────────────────────────────► S3 / CDN
```

1. Client requests the file via your API — never expose the S3 bucket directly.
2. Server looks up the DB record by `fileId`.
3. If `status ≠ "clean"`, return 404 or a `{ status: "pending" }` response.
4. If clean, generate a fresh presigned GET URL from `fileKey` + `bucket`:

```python
presigned_url = s3.generate_presigned_url(
    "get_object",
    Params={"Bucket": "prod-bucket", "Key": file_key},
    ExpiresIn=900,  # 15 minutes
)
```

5. Return the URL to the client. Client fetches directly from S3 or via CloudFront CDN (which caches the response at the edge).

## 7. Why Store the Key, Not the URL

| | Object key | Presigned URL |
|---|---|---|
| Lifetime | Permanent | Expires (minutes–hours) |
| Bucket migration | Update one config value | Regenerate all stored URLs |
| Access control | Enforced at read time | Baked in at generation time |
| Storage cost | Tiny string | Longer string, stale immediately |

Storing the key means you generate a fresh, properly-scoped URL on every request. This lets you:

- Rotate buckets without touching the DB
- Enforce per-user, per-request access checks
- Adjust TTL based on content sensitivity

## 8. Complete End-to-End Flow

```
Upload:
  Client → Server: "I want to upload avatar.jpg (10 KB)"
  Server → DB: INSERT (fileKey, bucket=quarantine, status=pending)
  Server → Client: presigned PUT URL (expires 5 min)
  Client → S3 quarantine: PUT file (direct, no server bandwidth)
  S3 → Lambda: "object created: uploads/user42/uuid.jpg"

Scan:
  Lambda → S3 quarantine: download file
  Lambda: run AV scan
  [clean]  Lambda → S3 prod: copy object
           Lambda → S3 quarantine: delete object
           Lambda → DB: UPDATE status=clean, bucket=prod
  [dirty]  Lambda → S3 quarantine: delete object
           Lambda → DB: UPDATE status=rejected
           Lambda → notification service: alert user

Download:
  Client → Server: GET /files/{fileId}
  Server → DB: SELECT (fileKey, bucket, status)
  status ≠ clean → 404
  status = clean →
    Server: generate presigned GET URL (fileKey, bucket=prod, TTL=15min)
    Server → Client: presigned URL
    Client → S3 / CDN: GET file bytes
```
