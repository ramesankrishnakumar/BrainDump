Database Choice — Interview Scenarios

Based on guide 5 · RDBMS, NoSQL, ACID, and BASE · Practice sessions

The meta-rule: name the database, tie it to one specific property that matters for this domain, state the consequence of getting it wrong. One sentence unprompted. Expand only when probed.

Three questions to run before answering: What must be correct? What must be fast? What can be stale?

Scenario 1 Taxi ride payment processing Easy

You've just described a system that processes payments at the end of each taxi ride — charge the rider, credit the driver, log the transaction.

Interviewer
"What database are you using here and why?"
Model answer + analysis
"Postgres — payments need atomic multi-step transactions. A charge and a ledger entry can't partially succeed."

Why this works

  • Names a specific database, not a category.
  • Atomicity is the right ACID property to cite here — not consistency, not isolation. If charge succeeds but ledger write fails, money disappears.
  • Gives a concrete consequence ("can't partially succeed") rather than reciting a definition.

Common mistakes

What candidates sayThe problem
"I need ACID consistency" Consistency is the weakest letter in ACID for interviews. The real risk is atomicity — partial writes.
"Relational database because there are relationships" NoSQL can model relationships too. This is the yellow flag the interviewer is watching for.
"SQL because I know it well" That's a personal preference, not a requirement-driven reason. Fine to say at the end, not as the reason.
Pattern: when money, inventory, or permissions are involved → lead with atomicity or transactions, name Postgres or MySQL.
Scenario 2 Social media activity feed Medium

You're designing the feed that shows each user the 50 most recent events from people they follow. Reads are very high volume. A post appearing 1–2 seconds late is acceptable.

Interviewer
"Same question — what are you storing the feed in?"
Model answer + analysis
"Cassandra — the feed access pattern is always 'latest N events by user_id', which maps directly to a wide-column partition. Slight staleness is fine here so I don't need ACID."

Why not Postgres here

Postgres can answer this query. The problem is at scale. "JOIN users, follows, posts WHERE follower = X ORDER BY created_at LIMIT 50" touches multiple tables on every feed load for millions of users. That's what breaks.

The feed read path is latency-sensitive with a completely fixed, known access pattern. That profile calls for a store modeled around that query.

The tell that reveals the wrong instinct

If your justification for a database is "I'll join two tables" — pause. Joins are an implementation detail, not a requirement. Ask whether that join runs on every request at scale.

Requirements breakdown

RequirementImplication
Must be fast at scaleSingle-partition read, no joins
Access pattern is fixedNo need for flexible SQL queries
Staleness of ~1s is fineEventual consistency (BASE) is acceptable
No financial correctnessACID not required
Pattern: fixed known access pattern + high read volume + staleness acceptable → wide-column store (Cassandra, DynamoDB). Model your query first, then pick the store.

Concept: What a wide-column store actually looks like

A wide-column row has three parts: a partition key (which node), a clustering key (order within that node), and sparse columns (different rows can have different columns). 

partition_key │ clustering_key       │ event_type │ content       │ liked_post_id
──────────────┼──────────────────────┼────────────┼───────────────┼──────────────
user1         │ 2026-05-24 10:00:00  │ post       │ "hello world" │ —
user1         │ 2026-05-24 09:00:00  │ like       │ —             │ 123
user1         │ 2026-05-24 08:00:00  │ follow     │ —             │ —
user2         │ 2026-05-24 11:00:00  │ post       │ "good morning"│ —

The query WHERE user_id = 'user1' ORDER BY ts DESC LIMIT 50 is a single sequential read within one partition — already sorted, no joins, no scatter across nodes.

The timestamp is the clustering key, not a separate dimension. It exists because you need multiple rows per user (one per event) and you need to know which 50 to return. The partition key routes you to the right node; the clustering key orders the rows once you're there.

Scenario 3 URL shortener Medium

Short code maps to a long URL. Writes are infrequent (someone creates a short link). Reads are very high volume (anyone clicking a link triggers a lookup). No transactions needed. Slight staleness on a newly created link is acceptable.

Interviewer
"What database are you using?"
Model answer + analysis
"DynamoDB or Redis — the access pattern is a pure key lookup: short code in, long URL out. No joins, no transactions, and a key-value store handles this at high read volume with low latency. I'd layer a CDN or Redis cache on top so most lookups never hit the database at all."

Why this is genuinely ambiguous — and that's okay

Postgres would also work here. The data model is simple, scale is manageable, and a keyed lookup with an index is fast. If the interviewer pushes back, saying "either works, I'm going DynamoDB because the access pattern is a pure key lookup at high read volume and I want to avoid operational overhead" is a strong, honest answer.

The skill being tested is whether you can justify the choice with requirements — not whether you pick the "correct" answer.

Requirements breakdown

RequirementImplication
Pure key lookupKey-value store is a natural fit
Very high read volumeCache layer is mandatory regardless of DB choice
No transactionsACID not required
Infrequent writesWrite throughput is not the bottleneck
Slight staleness on new links OKEventual consistency acceptable

The cache observation matters

Saying "I'd cache this" unprompted signals maturity. Most URL shortener traffic hits a small set of popular links. A CDN or Redis cache means the database choice barely matters for the hot path — almost all reads are served from cache.

Pattern: pure key lookup + no transactions + high reads → key-value store. Add caching observation unprompted — it shows you're thinking about the actual bottleneck.
Scenario 4 E-commerce checkout (polyglot) Hard

You're designing an e-commerce system. It has: order management, a product catalog (10M SKUs with varying attributes per category), a product search page, and user sessions.

Interviewer
"Walk me through your storage choices."
Model answer + analysis
"I'd use different stores for different jobs. Orders and payments go in Postgres — they need ACID transactions across order, payment, and inventory tables. The product catalog goes in a document store like MongoDB or DynamoDB — product attributes vary by category so a flexible schema fits better than forcing everything into fixed columns. Search goes to Elasticsearch — users need ranked full-text search with filters, which a document DB can't do well. Sessions go in Redis — pure key lookup, needs sub-millisecond latency, and it's fine if a session expires."

What this answer demonstrates

  • Polyglot persistence — different stores for different jobs, not one database for everything.
  • Each choice is justified by a specific property: transactions, schema flexibility, text ranking, latency.
  • No store is over-extended into a role it's bad at.

The trap: one database to rule them all

Candidates often try to pick one database and stretch it to cover every use case. The interviewer is watching for the recognition that search, sessions, flexible catalogs, and transactional orders have genuinely different requirements that are best served by different engines.

Storage allocation summary

ComponentStoreReason
Orders / paymentsPostgresACID transactions across related tables
Product catalogMongoDB / DynamoDBSchema varies per product category
Product searchElasticsearchRanked full-text search with filters
SessionsRedisPure key lookup, TTL expiry, sub-ms latency
Pattern: in complex systems, each data type has a different job. Name the store for each job and justify with one property. "One database should own each truth" — guide 5 §8.

Concept: DynamoDB is one engine — three modes

The "mode" is a data modeling decision, not a configuration switch. Same DynamoDB engine, different behavior based on key design.

ModeKey designBehaviorExample use
Key-valuePartition key onlyO(1) single item lookupSessions, feature flags
Wide-columnPK + sort key (timestamp)Range queries within partition, sorted rowsFeeds, order history, timelines
DocumentAny key + nested JSON itemFlexible per-item attributesProduct catalog, user profiles

Three ways to handle a secondary access pattern (in order of preference):

  1. Sort key prefix — encode item type into the SK (post#timestamp, like#timestamp). Use begins_with to filter. Cheap, co-located, no extra cost.
  2. Global Secondary Index (GSI) — DynamoDB maintains a separate index with a different PK. Allows a totally different access dimension. Costs extra read/write throughput.
  3. Filter expression — avoid on hot paths. Fetches the entire partition first, then discards non-matching rows in memory. Reduces response size, not read cost. 10,000 rows fetched to return 50 = 10,000 read units consumed.
Scenario 5 DynamoDB single-table: why not separate tables? Hard

You chose DynamoDB for the social feed. You've explained co-locating posts, likes, and follows in one table using sort key prefixes. The interviewer pushes back.

Interviewer
"Why not just have separate tables — posts, likes, follows? That feels cleaner."
Model answer + analysis
"DynamoDB has no joins. Separate tables mean multiple round trips — one per table — or expensive cross-table transactions. By encoding item type in the sort key prefix, all of user_123's data lives in one partition. I get one query, one round trip, and the prefix filter separates types cleanly."

Why single-table is idiomatic DynamoDB

In a relational DB, one table per entity is the right instinct — SQL joins bring them together cheaply at query time. DynamoDB has no joins. Every cross-table access is a separate network request. Single-table design is the DynamoDB community's answer to this constraint.

What the sort key prefix pattern looks like

PK: user_123  |  SK: post#2026-05-24T10:00   → { content: "hello" }
PK: user_123  |  SK: like#2026-05-24T09:00   → { liked_id: 456 }
PK: user_123  |  SK: follow#2026-05-24T08:00 → { followed: "user_99" }

Query all posts:   WHERE PK = 'user_123' AND SK begins_with 'post#'
Query everything:  WHERE PK = 'user_123' ORDER BY SK DESC LIMIT 50

Cassandra does the opposite — and that's fine too

Cassandra's convention is one physical table per access pattern. feed_by_user, posts_by_user, likes_by_user would all be separate tables. Neither approach is wrong — they reflect the different operational philosophies of the two engines.

DynamoDBCassandra
ConventionSingle-table, sort key prefixesOne table per query
Secondary accessGSIMaterialized views
Secondary indexesSafe (separate maintained index)Dangerous (queries every node)
Best fitManaged / serverlessOn-prem / high sustained writes
Pattern: DynamoDB = single-table, sort key encodes entity type. Cassandra = one table per query. Both avoid joins by modeling around the access pattern upfront.
Scenario 6 Cassandra vs DynamoDB — same requirements, how do you choose? Hard

You've said the feed needs a wide-column store. The interviewer knows both Cassandra and DynamoDB fit the access pattern.

Interviewer
"You mentioned Cassandra or DynamoDB. Both work. How do you actually pick one?"
Model answer + analysis
"For most product teams I'd default to DynamoDB — it's fully managed, scales automatically, and removes operational overhead. I'd lean toward Cassandra if we need on-prem deployment, very high sustained write throughput with fine-grained tuning, or we already have Cassandra expertise in-house. The data model philosophy differs too: DynamoDB leans single-table with sort key prefixes; Cassandra leans one table per query. Either works, but the team needs to commit to one modeling convention."

Decision factors

FactorLean DynamoDBLean Cassandra
InfrastructureCloud-managed, serverlessOn-prem or self-managed
Write throughputHigh but auto-scaledVery high, sustained, tunable
Ops overheadNear zeroSignificant (compaction, repair, tuning)
Data modelingSingle-table, GSIOne-table-per-query, materialized views
Cost modelPer request (unpredictable at spikes)Provisioned capacity (predictable)

What the interviewer is really testing

They want to see that you can distinguish the two beyond "both are NoSQL wide-column stores." Knowing the operational philosophy difference (single-table vs one-table-per-query) and the secondary index risk in Cassandra signals real familiarity.

Pattern: when requirements are ambiguous between two stores, justify on operational fit and team context — not just data model. "Either works; I'd pick X because…" is a strong, honest answer.
Scenario 7 Fashion retailer product catalog Medium

A fashion retailer with 50 million SKUs. Jeans have waist and length. Shoes have size, width, and style. Electronics have voltage and wattage. Every category has different attributes.

Interviewer
"How are you storing the product catalog?"
Model answer + analysis
"MongoDB or DynamoDB in document mode — product attributes vary structurally by category. Forcing this into a relational schema means either sparse nullable columns across hundreds of attributes, or an EAV table which is slow and hard to query. A document store handles per-category schemas naturally, and the access pattern is a simple key lookup by product ID."

Why relational breaks here

A single products table with every possible attribute across every category would have hundreds of columns, most null for any given row. The alternative — Entity-Attribute-Value (EAV) — stores attributes as rows, but then filtering by waist=32 becomes a painful pivot query. Both patterns are painful.

What EAV looks like (and why it's bad)

product_id | attribute   | value
1001       | waist       | 32
1001       | length      | 30
1001       | color       | blue
-- To filter: WHERE attribute='waist' AND value='32' AND attribute='color' AND value='blue'
-- This requires self-joining the table. Slow, hard to index.

Requirements breakdown

RequirementImplication
Schema varies by categoryDocument store handles flexible per-item attributes
Access is usually by product IDSimple key lookup — no complex joins needed
Search (color, size, price) is separateElasticsearch sits in front for filtered search
No financial transactionsACID not required
Pattern: variable schema per record → document store. If filtered search is also needed → document store as source of truth, Elasticsearch as the search layer.
