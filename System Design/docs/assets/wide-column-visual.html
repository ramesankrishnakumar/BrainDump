Wide-Column Database — Visual Guide

A wide-column store is not “a table with many SQL columns.” It is a sparse, partitioned grid: rows grouped by partition key, sorted by clustering key, with column families that grow horizontally over time.

What “wide” really means

Think of one partition (e.g. all events for user_123) as a row that gets wider over time as you append events — without defining 10,000 SQL columns upfront. Empty cells cost almost nothing (sparse).

Partition key — which machine / shard owns this data Clustering key — sort order inside the partition Columns — actual field values (can differ per row)

Mental model: Wide-column = spreadsheet per partition key, time grows to the right (new rows), each row can have different columns filled in.

Your exact question: one row, or many?

Common trap: “a feed is user1 → feed, so why split into time1, time2?” Because a feed is not one value — it’s an ordered log of many events that grows forever. Wide-column uses two different keys to do two different jobs:

Partition key = user1

Which machine stores the data. Co-locates all of user1’s events on one shard. This is your “user1 → feed” intuition — and it’s correct, at the partition level.

Clustering key = time1, time2…

Sort order + identity of each row inside the partition. Each event is its own row, kept physically sorted by time so “last 50” is one sequential read.

So the mapping is really:

user1
→
[ event@time1, event@time2, event@time3, … ]
one partition, many sorted rows — grows forever as events append

Model A vs Model B (both you proposed):

Model A — fixed wide row

user1, time1, col1, col2, … col50
user1, time2, col1, col2, … col50

This is only the logical CQL view. Misleading — it implies fixed, pre-allocated columns. Real storage is not like this.

Model B — row → present columns only ✓

user1, time1 → {type:like, target:post_9f2}
user1, time2 → {type:comment, target:post_7aa, text:"Nice"}

Closer to physical reality. Each row stores only columns that exist — a like has no text cell; it simply isn’t on disk (sparse). Cassandra even stores each as (user1, time1, colname) → value.

Why many rows, not one blob?

  • Append is cheap — new event = write one new row, never touch old data.
  • Read is a range scan — “last 50” = first 50 rows of an already-sorted partition. One seek.
  • One blob would be terrible — every new event would rewrite the whole feed (contention + unbounded row).

One line: one partition per user (the “user1 → feed” part), many rows per partition (one per event, sorted by time). Model B, not Model A.

Notice: text is empty for “like” and “follow” — in SQL you still pay for NULL columns across all rows; in wide-column, missing cells are cheap (sparse).

Relational table — fixed schema, all rows same columns

Every row must fit the same columns. Joins across tables are native. Great for correctness; harder to scale writes globally.

Query: SELECT * FROM user_events WHERE user_id = 'user_123' ORDER BY event_time DESC LIMIT 50;
Works, but at billions of rows you need indexes + sharding; cross-shard queries get expensive.

Document store — one JSON blob per record

Flexible nested shape. Great when each record is a self-contained object (profile, product). Less ideal for “millions of tiny events per user” unless you embed arrays (which grow huge).

{ "_id": "evt_001", "user_id": "user_123", "event_time": "2026-05-24T14:02:01Z", "type": "like", "target": "post_9f2" }
{ "_id": "evt_002", "user_id": "user_123", "event_time": "2026-05-24T13:58:44Z", "type": "comment", "target": "post_7aa", "text": "Nice shot!" }
Query: db.user_events.find({ user_id: "user_123" }).sort({ event_time: -1 }).limit(50)
Each event = separate document. 10M events = 10M documents (heavy vs one wide partition).

Key-value store — one key → one opaque value

Fastest lookup by exact key. No native “range of timestamps inside a user.” You must design the key carefully or store everything in one blob.

user_123:feed
→
Serialized list of 50 events (JSON / protobuf)
Update = read entire blob, append, write back
user_123:evt:2026-05-24T14:02:01
→
{ type: "like", target: "post_9f2" }
GET user_123:feed — O(1) but no efficient “events between May 20–24” unless you designed many keys upfront.

Same use case, four shapes

Store: “last 50 events for user_123” at massive scale (billions of events/day)

Wide-Column ✓ best fit

Partition by user_id, sort by time. Append-only writes on one shard. Range queries inside partition. Sparse columns.

Examples: Cassandra, HBase, Bigtable

Document △ OK

One doc per event is simple but many small docs. Embedding events in user doc = huge document, painful updates.

Examples: MongoDB, Couchbase

Key-Value △ with tricks

Whole feed as one value = fast read, bad for partial updates. Per-event keys = no cheap time-range scan.

Examples: Redis, DynamoDB (also wide-column-ish)

Relational ✓ until scale

Fixed schema, ACID, SQL. At extreme write volume, sharding + hot rows become the bottleneck.

Examples: PostgreSQL, MySQL
